NOMBRE DE APLICACIÓN TIME FLOW
1. Caso de uso: Importar Datos
Actor: Usuario
Precondición: El usuario ha accedido al sistema.
Poscondición: El usuario tiene los datos importados en el sistema.
Flujo de eventos básico:
1.	El caso de uso comienza cuando el usuario selecciona el tipo de fuente de datos (Archivo o Base de datos).
2.	Si es un archivo, el usuario selecciona el archivo deseado (CSV, Excel, JSON).
3.	Si es una base de datos, el usuario selecciona la base de datos a la que conectarse.
4.	El sistema importa los datos y muestra una vista preliminar para verificar la carga correcta.
5.	Si el usuario no dispone de datos, el sistema le ofrece la opción de generar automáticamente una serie simulada con un mínimo de 100 y un máximo de 1000 observaciones y el caso de uso finaliza.
Flujos de eventos alternativos:
• Archivo no válido:
En el paso 2, si el archivo seleccionado no es válido (por ejemplo, formato incorrecto o archivo corrupto), el sistema muestra un mensaje de error y permite al usuario seleccionar otro archivo o cancelarlo, luego el caso de uso finaliza.
• Conexión fallida a la base de datos:
En el paso 3, si el sistema no puede conectar con la base de datos seleccionada (por ejemplo, credenciales incorrectas o error de red), el sistema muestra un mensaje de error y permite al usuario intentar de nuevo o cancelar la operación, luego el caso de uso finaliza.
2. Caso de uso: Conectar a fuentes de datos
Actor: Usuario
Precondición: El usuario ha accedido al sistema.
Poscondición: El usuario está conectado a una fuente de datos en tiempo real.
Flujo de eventos básico:
1.	El caso de uso comienza cuando el usuario selecciona la opción para conectarse a una fuente de datos en tiempo real desde la interfaz.
2.	El sistema solicita las credenciales o parámetros necesarios para la conexión (como URL, claves de autenticación o tokens).
3.	El sistema establece la conexión a la fuente de datos en tiempo real a través de Streamlit.
4.	El sistema recibe y muestra los datos en tiempo real para su procesamiento y análisis.
Flujos de eventos alternativos:
• Código de usuario o contraseña erradas:
En el paso 2, si el código de usuario o contraseña son incorrectos, el sistema muestra un mensaje y solicita las credenciales nuevamente.
• Fuente de datos no disponible:
En el paso 3, si la fuente de datos no está disponible, el sistema muestra un mensaje de error y el caso de uso finaliza.
3. Caso de uso: Detectar y manejar valores atípicos
Actor: Usuario
Precondición: El usuario ha cargado una serie temporal de datos.
Poscondición: Los valores atípicos han sido manejados correctamente.
Flujo de eventos básico:
1.	El caso de uso comienza cuando el sistema recibe una serie de tiempo para su análisis.
2.	El sistema detecta automáticamente los valores atípicos dentro de la serie de tiempo.
3.	El sistema muestra una lista de los valores atípicos detectados al usuario.
4.	El usuario selecciona una acción a realizar (por ejemplo, reemplazar con la media, eliminar o aplicar un tratamiento específico).
5.	El sistema aplica el tratamiento seleccionado y actualiza la serie de tiempo.
Flujos de eventos alternativos:
• No se detectan valores atípicos:
En el paso 3, si el sistema no puede identificar un valor atípico, muestra un mensaje indicando que no se encontraron valores atípicos y el caso de uso finaliza.
4. Caso de uso: Interpolar y completar datos faltantes
Actor: Usuario
Precondición: El usuario ha cargado una serie temporal con datos faltantes.
Poscondición: Los datos faltantes han sido interpolados o completados.
Flujo de eventos básico:
1.	El caso de uso comienza cuando el sistema detecta que existen valores faltantes en la serie temporal.
2.	El sistema solicita al usuario que seleccione el método de interpolación deseado (por ejemplo, lineal, polinomial, spline).
3.	El sistema aplica el método de interpolación seleccionado a los datos faltantes.
4.	El sistema muestra la serie temporal completa con los valores interpolados.
Flujos de eventos alternativos:
• Método de interpolación no aplicable:
En el paso 2, si el método de interpolación seleccionado no es aplicable a los datos, el sistema muestra un mensaje de error y solicita seleccionar otro método.

8. Casos de uso: Revisar componentes de series de tiempo
Actor: Usuario

Poscondición: El usuario ha revisado los componentes de la serie de tiempo.

9. Casos de uso: Calculadora ACF y PACF
Actor: Usuario
Precondición: El usuario ha cargado una serie de tiempo en el sistema.
Poscondición: El usuario ha obtenido las funciones ACF y PACF de la serie.

1.	El caso de uso comienza cuando el usuario selecciona la serie de tiempo que desea analizar.

3.	El sistema presenta los resultados de ACF y PACF en forma de gráfico.
4.	El usuario interpreta los gráficos y ajusta los modelos según sea necesario y el caso de uso finaliza.
Flujos de eventos alternativos:
• Cálculo no realizado:
En el paso 2, si el sistema no puede calcular ACF y PACF, muestra un mensaje de error y permite al usuario intentar de nuevo y el caso de uso finaliza.
10. Casos de uso: Identificador de patrones temporales
Actor: Usuario
Precondición: El usuario ha cargado una serie de tiempo en el sistema.
Poscondición: El sistema ha identificado patrones temporales y ha recomendado un modelo adecuado.
Flujo de eventos básico:
1.	El caso de uso comienza cuando el usuario selecciona la serie de tiempo que desea analizar.
2.	El sistema realiza pruebas de estacionariedad (por ejemplo, Dickey-Fuller) para evaluar la naturaleza de la serie.
3.	El sistema identifica los patrones temporales (tendencia, estacionalidad, irregularidad) en la serie.
4.	El sistema presenta los patrones identificados al usuario.
5.	El sistema recomienda el tipo de modelo a aplicar según los patrones encontrados y el caso de uso finaliza.
Flujos de eventos alternativos:
• Prueba de estacionariedad fallida:
En el paso 2, si la prueba de estacionariedad no se puede realizar, el sistema muestra un mensaje de error y permite al usuario intentar de nuevo y el caso de uso finaliza.
11. Casos de uso: Aplicar modelo SARIMA
Actor: Usuario
Precondición: El usuario ha identificado patrones temporales en la serie de tiempo.
Poscondición: El modelo SARIMA ha sido aplicado y los resultados se han mostrado.
Flujo de eventos básico:
1.	El caso de uso comienza cuando el usuario selecciona la opción de modelado.
2.	El sistema solicita al usuario definir los parámetros estacionales y no estacionales del modelo SARIMA (p, d, q) y (P, D, Q, s).
3.	El sistema entrena el modelo SARIMA sobre la serie temporal proporcionada.
4.	El sistema muestra los resultados: métricas de ajuste, coeficientes estimados y gráfico del pronóstico.
5.	El usuario puede guardar el modelo o realizar ajustes y el caso de uso finaliza.
Flujos de eventos alternativos:
• Parámetros no definidos correctamente:
En el paso 2, si los parámetros no son definidos correctamente, el sistema muestra un mensaje de error y solicita al usuario que los ajuste y el caso de uso finaliza.
12. Casos de uso: Validación del modelo SARIMA
Actor: Usuario
Precondición: El modelo SARIMA ha sido aplicado.
Poscondición: El usuario ha validado el modelo SARIMA.
Flujo de eventos básico:
1.	El caso de uso comienza después de aplicar el modelo SARIMA.
2.	El sistema evalúa los residuos del modelo con pruebas como Ljung-Box, normalidad, independencia y heterocedasticidad.
3.	El sistema presenta los resultados de validación en forma de resumen y gráficos.
4.	El usuario interpreta la validez del modelo y decide si es adecuado y el caso de uso finaliza.
Flujos de eventos alternativos:
• Validación fallida:
En el paso 2, si la validación no es exitosa, el sistema muestra un mensaje de error y permite al usuario intentar de nuevo y el caso de uso finaliza.
